The Curse of Monads: Breaking the Spell

Have you heard of the Curse of Monads? It's a real, documented phenomenon in computer science. The legend says: once you finally understand what a monad is, you lose the ability to explain it to anyone else.

On the internet, you'll find all sorts of explanations involving burritos, spacesuits, or even toxic waste. But honestly, it's a remarkably simple concept. Today, we're going to break the curse.

Level 0: The Prerequisites

Before we touch the "M-word," let's make sure our foundation is solid. We need to look at three things: Functional Paradigms, Higher Order Functions, and Closures.

1. Data is Input, Not State

The easiest way to understand this is by comparing it with Object-Oriented Programming (OOP).

  • OOP: Data is state. You have an object O and you tell it O(x) -> O. "Object, please compute f(x) and update yourself." The object changes internally.
  • Functional: Data is input. y = f(x). You give x to a function, and it spits out y. Crucially, x stays exactly as it was. No state is changed; it's just mathematical purity.

2. Higher Order Functions

A first-order function is simple: y = f(x), where y is just an output. A Higher Order Function is a bit more meta: it can return a function as its output, or take a function as its input. It's functions all the way down.

3. Closures (Private Backpacks)

Closures are essentially functions with a private state or access to their environment. Think of them as the functional equivalent of "Private" variables in OOP.

function closure_factory()
    local x = 0
    return function()
        x = x + 3
        print("Progress: " .. x)
    end
end

local step_up = closure_factory()
step_up() -- Prints 3
step_up() -- Prints 6

You can't touch x from the outside. It's locked inside the closure's "memory."

The Monad: A Futuristic Package

Now, let's talk about what a monad actually is. The simplest explanation? A monad is a Wrapper for Data with a very specific set of rules.

Imagine a Factory that produces futuristic packages. You can put anything inside be it a dog, a cat, a string of text but only at the moment of manufacture.

This package is special. It will do anything you tell it to do to the contents, but it follows a specific workflow:

  1. You give the package an instruction: "Hey package, whatever is inside you, make it a cat."
  2. The package modifies the internal dog into a cat.
  3. The Magic: You don't get the same package back. You get a Brand New Package containing the cat.
The Teleportation Clause: If you say "Keep the package empty," it teleports the item somewhere else and remains empty. But once a package is empty, it loses the ability to follow further instructions. It just stays empty.

Monads in the Wild (Eager Edition)

In the C++ world, we call this std::optional<T>. It's the Maybe Monad.

std::optional<int> half(int x) {
    if (x % 2 == 0) return x / 2;
    return std::nullopt; // The "Empty" state
}

auto result = std::optional{40}
    .and_then(half)
    .and_then(half)
    .and_then(half);

// result will hold 5. If any step returned nullopt, 
// the rest of the chain would simply not execute.

In this scenario, the constructor is the Unit (putting data in the package) and and_then is the Bind function.

Lazy Monads: Doing Nothing as an Art

The examples above are eager—they execute immediately. But sometimes we want to be lazy. We want to define a chain of instructions and only execute them when we're ready.

The Lazy Monad (Lua)

Let's build a manufacturer for lazy packages that only compute when we call :eval().

-- Lazy Monad Factory
function Lazy(value_or_fn)
    local self = {}

    if type(value_or_fn) == "function" then
        self.thunk = value_or_fn
    else
        self.thunk = function() return value_or_fn end
    end

    function self:and_then(f)
        return Lazy(function()
            local v = self.thunk()
            if v ~= nil then
                -- f(v) returns a Lazy object; we evaluate it
                return f(v):eval()
            else
                return nil
            end
        end)
    end

    function self:eval()
        return self.thunk()
    end

    return self
end

-- Usage
local result = Lazy(40)
    :and_then(function(x) return Lazy(x / 2) end)
    :and_then(function(x) return Lazy(x / 2) end)
    :and_then(function(x) return Lazy(x / 2) end)

print(result:eval()) -- prints 5

And there it is. The curse is broken. You now understand Monads as effectively as a Haskeller, but with much better metaphors. Now, go forth and explain it to someone else. If you can.
